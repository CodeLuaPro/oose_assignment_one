2 Problem Description

Your application will be an experimental city planning tool.

It will work with a grid representing the land area of the city, where each grid square is one unit of land (on which a structure may-or-may-not be built). Grid squares have several characteristics:

  • A “terrain” type: either swampy, flat or rocky;
  • Any combination of “zoning rules”, including “heritage”, “height limit”, “flood risk”, and/or “contamination”, where:
    • A heritage listing is also accompanied by a construction material (“wood”, “stone” or “brick”).
    • A height limit is accompanied by a positive integer, representing a number of floors.
    • A flood risk is accompanied by a percentage, indicating the degree of risk.

2.1 Grid Data File

When your application first loads, it must read a data file given on the command-line.

How to accept command-line filenames in Java

If using Gradle, you can run:

$ ./gradlew run --args="thefile.txt"

The string "thefile.txt" will then be available as the first element of the args array parameter in your main() method:

public class MainClass
{
    public static void main(String[] args)
    {
        if(args.length != 1) { ... } // Error
        else
        {
            String filename = args[0];
            ...
        }
    }
    ...
}

How to accept any filename? There’s no trick to that, except to not manipulate it. For instance, don’t write "dir/" + args[0] (or something like it). Just take args[0] as-is.

The problem of supplying the correct filename is thus the user’s problem. If, when testing your program, you get a "No such file or directory" error, then the problem is probably that, as a user, you haven’t supplied the correct filename, or the correct path to the filename.

The data file lists out the contents of a grid, with terrain and zoning rules (but not structures). Here’s an example:

swampy,flood-risk=43.635280,heritage=wood
flat,heritage=wood
flat
flat,heritage=wood,contamination
swampy,flood-risk=41.534353,heritage=wood
flat
flat,heritage=stone
swampy,heritage=wood,flood-risk=34.581390,height-limit=24
rocky,heritage=brick
flat,heritage=stone
flat,height-limit=22

And there’s a visualisation of what the above file represents (with zoning rules heavily abbreviated):

Find oose2025s1_assignment1_datafile_generator.zip in the Assignments area on Blackboard, and run the script located in the bin/ directory. This will generate a random data file for testing purposes (named “grid-...”), and also show you a visualisation of it. You can change the size of the generated grid by passing command-line arguments; e.g.:

$ cd oose2025s1_assignment1_datafile_generator
$ bin/generate -H 6 -W 4

Your application will be expected to work with any of these auto-generated files!

Also find, on Blackboard, a selection of deliberately invalid data files. Your application must recognise these as invalid, and produce friendly, informative error messages.

Here are the technical details of the file format:

  • The first line contains the dimensions of the grid: height, then width (both positive integers).
  • Each subsequent line represents one grid square, in “row-major” order. Each line has a comma-separated list of fields.
  • The first field represents the terrain, and is "flat", "swampy" or "rocky".
  • The remaining fields (if any) represent zoning rules. Zoning rules may occur in any order, but there is at most one of each type, for a given grid square.
  • Zoning rules start with "heritage", "height-limit", "flood-risk" or "contamination".
  • Zoning rules, other than contamination, also contain a “=”, then a value.
    • For heritage, the value is either wood, stone or brick.
    • For height-limit, the value is a positive integer.
    • For flood-risk, the value is a real number in the range 0.0–100.0.
    • There is no value for contamination.

2.2 Menu

Your application must have a simple text-based menu system, with the following options (which the user is free to choose in any order!):

  • “Build Structure”;
  • “Build City”;
  • “Configure”;
  • “Quit”.

(The “Build” options do not actually modify the grid in any way. Think of them as testing what could be built.)

2.3 “Build Structure” Option

If/when the user chooses this option, the application must prompt for:

  • The location of a specific grid square, in terms of the row and column indexes (non-negative integers);
  • A number of floors (for the structure), a positive integer;
  • A foundation type: either "slab" or "stilts";
  • A construction material: "wood", "stone", "brick" or "concrete".

(You don’t specifically need to make the user enter these words. The important thing is to capture the information.)

Then, the application must report back:

  1. Whether the structure can be built;
  2. If not, why not;
  3. If so, how much it will cost.

To do this, it must apply various rules:

  • Regarding allowable structures:

    • A slab foundation cannot be built in a swamp;
    • A wooden structure cannot be built in a swamp;
    • If a heritage zoning rule applies, the structure cannot be built if the construction material is different from the heritage rule (e.g., wood vs. concrete).
    • If a height-limit zoning rule applies, the structure cannot be higher than (have more floors than) the specified limit;
    • If a flood-risk zoning rule applies, the structure must have at least two floors.

  • Regarding construction cost:

    • A wooden structure costs $10,000 per floor.
    • A stone structure costs $50,000 per floor.
    • A brick structure costs $30,000 per floor.
    • A concrete structure costs $20,000 per floor.
    • Building on a swamp costs an extra $20,000 per floor.
    • Building on rocky terrain costs an extra amount of $50,000 (irrespective of the number of floors).
    • Building on contaminated land multiplies the overall cost by 1.5.
    • Building in a flood-risk zone multiplies the cost by 1+risk50.

The multiplicative factors apply after the others.

2.4 “Build City” Option

If/when the user chooses this option, the application attempts to build structures on all grid squares.

It uses one of several approaches for doing this, but it does not ask which approach to use at this point. Instead, the approach to use is set via the “Configure” Option.

The different approaches are:

  1. “Uniform”: Prompt the user for a type of structure (the same as in Section 2.3), and try to build that on every grid square.

  2. “Random”: For each grid square, choose a type of structure at random. That is, select a random number of floors, a random foundation type and a random construction material.

  3. “Central”: The type of structure to be built is determined by the distance from the centre of the grid.

    Distance is defined as distance=(i−height−12)2+(j−width−12)2, where (i,j) are the row and column indexes of the grid square. Then:

    • The number of floors should be nFloors=round(1+20distance+1).
    • The construction material should be:
      • “concrete”, if distance≤2;
      • “brick”, if 2<distance≤4;
      • “stone”, if 4<distance≤6;
      • “wood” otherwise.
    • The foundations are always slabs for this purpose.

Regardless of approach, the application reports three things:

  1. The total number of structures able to be built (not just the total number of grid squares, of course);
  2. The total cost (of structures able to be built);
  3. An on-screen grid showing where the structures could be built.

The grid need not be sophisticated. At a minimum, it could simply look like this (for a 3x5 grid), where an “X” represents a successfully-built structure at a given location:

XX X
 XXXX
X XX

You are free to make this prettier and/or more informative, if you like.

2.5 “Configure” Option

If/when the user chooses this option, the application must prompt for a particular city-construction approach to use: “uniform”, “random” or “central”. This choice is then used subsequently when the user next chooses the “Build City” option.

The default approach is “random”. Hence, if the user selects the “Build City” without first selecting “Configure”, a random city will be built.
